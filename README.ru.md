## Почему это существует

IBM AP-101B летал на каждом Space Shuttle с STS-1 (1981) до конца 1980-х.
Его память была не кремниевой — это был **ферритовый сердечник**: тысячи
крошечных магнитных колец, вручную прошитых медной проволокой в плоскости
памяти (Core Memory Planes). CPU вмещал 80K 32-битных слов ферритового
хранилища; IOP — ещё 24K. Полётное ПО (PASS + BFS) умещалось примерно
в **256 КБ полезной ферритовой памяти**.

Этот тестовый фреймворк чтит инженеров, спроектировавших полётное ПО
Shuttle под это ограничение — подвиг системной инженерии, удерживавший
аппарат весом 2 000 тонн на орбите 30 лет.

Переход с феррита на кремний не был одномоментным. Разные орбитеры
модернизировались во время планового обслуживания (OMDP). Последний полёт
AP-101B — **STS-40** (Columbia, июнь 1991). Первый полёт AP-101S —
**STS-37** (Atlantis, апрель 1991). Два года флот Shuttle летал смешанным —
часть орбитеров на феррите, часть на КМОП.

Когда NASA перешло на полупроводниковую память в AP-101S, обнаружилось то,
с чем инженеры ферритовой эпохи никогда не сталкивались: КМОП SRAM уязвима
к космическим лучам. Магнитные сердечники естественно радиационно-стойки.
Кремний — нет. На STS-37 датчики зафиксировали первые реальные Single Event
Upsets — спонтанные перевороты битов от попадания высокоэнергетических частиц
в ячейки памяти. NASA пришлось активировать аппаратную логику ECC посреди
миссии.

**Тест SEU в этом фреймворке (AP101B-CORE-05) воспроизводит ровно ту
физическую аномалию, что проявилась на STS-37.** Феррит был невосприимчив.
КМОП потребовал ECC. Дисциплина B-модели — не про железо. Она про мышление:
ваш код должен выживать там, где кремний не может.

**Это не тест, проверяющий, работает ли ваш код в 256 КБ памяти.**
Это дисциплина. Она построена как инструмент внутреннего аудита для
**[Clarus PACS](https://github.com/clarus-pacs/clarus)** — сверхкомпактного
DICOMweb-сервера медицинской визуализации — чтобы проверять, что наш
продакшн-код на Rust не утекает heap-аллокациями в горячих путях,
не содержит неопределённого поведения в unsafe-блоках и обнаруживает
однобитовые ошибки космических лучей в критических структурах данных.

Подробная история Shuttle и перехода с феррита на КМОП:
**[HISTORY.md](HISTORY.md)** (русская версия: **[HISTORY.ru.md](HISTORY.ru.md)**).

### Что `cargo test -p ap101b-core` значит для [Clarus](https://github.com/clarus-pacs/clarus)

Когда мы впервые применили дисциплину AP-101B к Clarus Core, мы нашли
**128 000 ненужных heap-аллокаций на одно КТ-исследование** в конвейере
2D-дедупликации блоков. Исправление — хранение сырых `[u8; 32]` хешей
вместо hex-кодированных `String` — устранило шторм аллокаций.

В сочетании с дополнительными исправлениями (устранение VR-строк в DICOM-парсере,
zero-copy извлечение пикселей), heap-оборот на исследование упал
с **~22 МБ до ~5 МБ**. На бюджетном ARM-контроллере с 256 МБ ОЗУ —
типичном для региональной больницы — это разница между 3 одновременными
STOW-загрузками (затем OOM-kill) и **6+ стабильными**.

На современном сервере с 64 ГБ ОЗУ то же исправление означает на 128 000
меньше вызовов аллокатора на КТ-исследование, меньше фрагментации кучи,
меньше блокировок аллокатора при конкурентной нагрузке и более гладкие
latency-хвосты. Дисциплина масштабируется в обе стороны: сохраняет жизнь
бюджетному железу и ускоряет мощное.

> **Результат — не просто оптимизация кода. Это снижение риска повторного
> облучения пациента.** Когда встроенный контроллер томографа исчерпывает
> память, исследование теряется. Пациента вызывают на повторное сканирование —
> дополнительная доза, дополнительные расходы, дополнительное время.
> Дисциплина AP-101B устранила этот режим отказа. `Definitum semel.`

## Что делает

`ferrite-core` — библиотека `no_std` Rust, предоставляющая:

- **`FerriteCell<T>`** — ячейка памяти с контролируемым доступом и явным
  контрактом безопасности. Безопасное чтение для Copy-типов. Небезопасная
  запись с volatile-семантикой. Инжекция космического луча для SEU-тестирования.
- **Структурный аудит** — проверки геометрии на этапе компиляции
  (`size_of`, `align_of`, отсутствие скрытого padding) через макрос
  `assert_no_padding!` и рантайм-отчёт `GeometryReport`.
- **Контрольные суммы** — трейт `Checksum` с реализациями `Crc16`, `Crc32`,
  `XorFold`. Только стек. Ноль аллокаций.
- **IBM PASS CRT телеметрия** — декоратор `IbmCrt` с историческими
  идентификаторами сообщений `AP101B-CORE-`.

`ferrite-testkit` (требует std) предоставляет:

- **`TestAllocator`** — обёртка глобального аллокатора для проверки
  отсутствия heap-аллокаций в коде под тестом.
- **SEU-стратегии** — proptest-генераторы для симуляции single-event upset.

## Быстрый старт

Добавьте в `Cargo.toml`:

```toml
[dependencies]
ferrite-core = { git = "https://github.com/clicker71/ap101", package = "ferrite-core" }

[dev-dependencies]
ferrite-testkit = { git = "https://github.com/clicker71/ap101", package = "ferrite-testkit" }
```

## Соглашение об именах

| Префикс ID | Модель | Технология памяти | Разрядность | Объём | Эпоха |
|:---|:---|:---|:---|:---|:---|
| `AP101B-CORE-` | IBM AP-101**B** | Ферритовый сердечник | 32-бит | ~416 КБ физ., ~256 КБ полез. | 1981–1990. Последний полёт: STS-40 (1991) |
| `AP101S-CMOS-` | IBM AP-101**S** | КМОП SRAM (батарея) + DRAM/ECC | 32-бит | 256K слов (1 МБ) | 1991–2011. Первый полёт: STS-37 (1991). Первое обнаружение SEU. |

### Фильтрация тестов

```bash
cargo test -p ap101b-core    # Только тесты AP-101B (феррит)
cargo test -p ap101s-cmos    # Только тесты AP-101S (КМОП)
cargo test --workspace       # ВСЕ тесты (B + S + core + testkit)
```
