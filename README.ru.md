# AP-101B Ferrite Discipline

Инструмент контроля целостности памяти для окружений без стандартной библиотеки (no_std), созданный по образцу бортового компьютера IBM AP-101B космических челноков. Включает контрольные суммы CRC-32, имитацию одиночных сбоев и проверку плотной упаковки данных.

## Мнимое могущество высокоуровневых языков

Программисты привыкли считать, что память бесконечна, а абстракции ничего не
стоят. Это опасное заблуждение.

Когда пишут `print("Hello World")` на Python или вкладывают JSON-объект в
JavaScript — видят одну строку. А под ней — лавина. Среда исполнения
разворачивает контекст кучи, аллокатор вырезает блок виртуального адресного
пространства, блокируя потоки. Кэш-линии загрязняются, центральный процессор
простаивает десятки и сотни тактов, ожидая данные из медленной оперативной
памяти.

В облаке с безлимитной корпоративной картой эти расходы незаметны.
В критических системах за них платят жизнями.

**ap101** возвращает мышление уровня машинных команд в современный код на Rust.
Это автоматический сторож, заставляющий уважать железо.

## Четыре столпа

Это не симулятор старой физики. Это контракт качества для промышленного кода.
Ограничение в 256 КБ выбрано не как физический предел, а как граница
дисциплины, за которой действуют четыре необсуждаемых правила:

### 1. Плотная работа с кэшем — стена L1/L2

64 ГБ оперативной памяти DDR5 — обман. Сверхбыстрые кэши L1 и L2 внутри
процессора всё так же крохотны: от 32 до 512 КБ.

**Ловушка.** Если в нагруженном участке данные прыгают по куче через ссылки —
`String`, `Vec`, `Box` — вы упираетесь в стену кэша. Процессор встаёт и ждёт
шину памяти.

**Выход.** `ferrite-core` требует непрерывной плотной укладки данных без
скрытых пустот. Весь рабочий набор помещается в кэш-линии L1/L2. Код работает
на предельной скорости кремния.

### 2. Никаких паник во встраиваемых системах

В авионике, автомобильных блоках и медицинской технике необработанная нехватка
памяти означает катастрофический отказ.

**Ловушка.** Одно-единственное выделение временной строки внутри обработчика
прерывания под пиковой нагрузкой — и ядро уходит в panic.

**Выход.** Этот инструмент работает как жёсткий фильтр непрерывной интеграции.
Он отлавливает выделение памяти во время выполнения и несовпадение размеров
структур, не давая случайно испортить кучу в системах жёсткого реального
времени.

### 3. Экономика облака и периферийных узлов

На пограничных устройствах и высоконагруженных серверах основная потеря
производительности — не вычисления, а непрерывное выделение и освобождение
памяти.

**Ловушка.** Миллионы обращений к куче в секунду дробят память, заставляя
сборщик мусора или менеджер потоков входить в клинч.

**Выход.** Критические участки переписываются на стековые и линейные
структуры. Нагрузка на процессор со стороны системы падает до нуля. Меньше
облачных машин — меньше расходы.

### 4. Уязвимость кремния при 3 нм

Бортовой компьютер шаттла AP-101B использовал магнитную ферритовую память,
невосприимчивую к космическим лучам на уровне физики. Современный транзистор
тоньше 10 нм настолько мал, что поймать переворот бита можно от обычного
атмосферного нейтрона — прямо здесь, на Земле.

**Ловушка.** Высотные IoT-устройства, автомобильные контроллеры, медицинские
приборы страдают от скрытых повреждений данных, от которых стандартный
компилятор не спасает.

**Выход.** Встроенная имитация одиночного сбоя заставляет писать и проверять
программные средства отказоустойчивости (CRC-32, XorFold, ECC), гарантирующие
целостность данных, когда кремний подводит.

**Подробная история** — переход шаттлов с ферритовых сердечников на
полупроводники, STS-37, STS-40 и почему это важно сегодня:
**[История полётов AP-101 STS](./HISTORY.ru.md)** (English: **[AP-101 STS Flight History](./HISTORY.md)**).

## Доказательство делом: Clarus PACS

Инструмент родился из **[Clarus PACS](https://github.com/clarus-pacs/clarus)** —
компактного DICOMweb-сервера для передачи и хранения медицинских изображений.

Первое же применение дисциплины AP-101B к подсистеме поиска дубликатов в
Clarus Core обнаружило **128 000 лишних выделений памяти на одно
КТ-исследование** — только из-за того, что бинарные хэши SHA-256
преобразовывали в шестнадцатеричные строки `String`.

После внедрения правил ap101:

- Заменили строки на обычные массивы `[u8; 32]`.
- Убрали лавину выделений при разборе заголовков DICOM.
- Научились отдавать пиксельные данные без копирования (`&[u8]` вместо `Vec<u8>`).

| Показатель | Было (String) | Стало ([u8; 32]) |
|:---|---:|---:|
| Выделений памяти на одно КТ-исследование | 128 000 | 0 |
| Оборот кучи на исследование | ~22 МБ | ~5 МБ |
| Одновременных загрузок снимков (256 МБ ОЗУ) | 3 → отказ | 6+ (устойчиво) |
| Размер ChunkRecord | 40 Б (+ куча) | 40 Б (стек) |
| Риск повторного облучения пациента | Есть | Устранён |

На сервере с 64 ГБ ОЗУ этот же подход означает на 128 000 меньше обращений
к аллокатору на каждое КТ-исследование, меньше фрагментации кучи, ниже
задержки под нагрузкой и более ровный график времени отклика. Дисциплина
работает в обе стороны: спасает слабое железо и ускоряет мощное.

> **Результат** — не просто быстрый код. Это устранённый риск повторно
> облучить человека. Когда встроенный компьютер томографа исчерпывает
> память — снимок теряется. Пациента вызывают на повторное сканирование:
> лишняя доза радиации, лишние расходы, потерянное время. Дисциплина
> AP-101B убрала этот сценарий полностью. `Definitum semel.`

Мы делимся этим инструментом, потому что на счету каждый байт — в космосе,
в медицине, в любой системе, где отказ недопустим.

## Состав

**ferrite-core** — библиотека Rust без стандартной библиотеки (no_std):

- **`FerriteCell<T>`** — ячейка памяти с явными правилами доступа. Чтение
  безопасно для простых типов. Запись — только через `unsafe`. Имеется
  встроенный имитатор попадания космической частицы.
- **Структурный аудит** — проверки на этапе компиляции: размер, выравнивание,
  отсутствие скрытых пустот (padding). Макрос `assert_no_padding!` и отчёт
  `GeometryReport`.
- **Контрольные суммы** — типаж `Checksum` с готовыми реализациями CRC-16,
  CRC-32 и XorFold. Только стек, без кучи.
- **Телеметрия IBM PASS CRT** — `IbmCrt` для вывода в стиле пультовой
  электронно-лучевой трубки шаттла. Префиксы сообщений `AP101B-CORE-`.

**ferrite-testkit** (требует стандартную библиотеку):

- **`TestAllocator`** — обёртка глобального аллокатора, не дающая
  проверяемому коду незаметно уйти в кучу.
- **Стратегии одиночных сбоев** — генераторы `proptest` для перебора
  битовых переворотов.

## Быстрый старт

В `Cargo.toml`:

```toml
[dependencies]
ferrite-core = { git = "https://github.com/clicker71/ap101", subdir = "codebase/ferrite-core" }

[dev-dependencies]
ferrite-testkit = { git = "https://github.com/clicker71/ap101", subdir = "codebase/ferrite-testkit" }
```

## Обозначения

| Префикс | Модель | Память | Разрядность | Объём | Время |
|:---|:---|:---|:---|:---|:---|
| `AP101B-CORE-` | AP-101B | Ферритовые сердечники | 32 бита | ~416 КБ общий, ~256 КБ под задачу | 1981–1990. Последний полёт: STS-40 (1991) |
| `AP101S-CMOS-` | AP-101S | КМОП-СОЗУ (батарея) + ДОЗУ/ECC | 32 бита | 256K слов (1 МБ) | 1991–2011. Первый полёт: STS-37 (1991). Первый зарегистрированный сбой. |

```bash
cargo test -p ap101b-core    # Феррит
cargo test -p ap101s-cmos    # КМОП
```
